# <a name="identity-and-device-access-configurations"></a>ID とデバイスのアクセス構成

この一連の記事は、推奨環境と定められた一連の条件付きのアクセス ポリシーの構成を実装することでエンタープライズ モビリティとセキュリティ製品をクラウド サービスにセキュリティで保護されたアクセスを構成する方法を説明し、関連する機能です。Azure AD アプリケーション プロキシを使用して他の SaaS のサービスとオンプレミス アプリケーションの発行に、Office 365 サービスを含め、Azure Active Directory と統合されているすべてのサービスへのアクセスを保護するためにこのガイドを使用できます。 

これらの推奨事項は、マイクロソフトのセキュリティで保護されたスコアと同様に[ユーザーは Azure AD のスコア](https://docs.microsoft.com/en-us/azure/active-directory/fundamentals/identity-secure-score)に揃えて配置され、これらの組織のスコアが増加します。これらの推奨事項も、これらの[5 つの手順の id インフラストラクチャのセキュリティ保護](https://docs.microsoft.com/en-us/azure/security/azure-ad-secure-steps)を実装できます。 

マイクロソフトでは、一部の組織ではある固有の環境の要件や複雑性を理解しています。これらの組織のいずれかの場合は、開始点としてこれらの推奨事項を使用します。ただし、ほとんどの組織は、規定に従って、これらの推奨事項を実装できます。 

## <a name="intended-audience"></a>対象ユーザー

これらの推奨事項は、エンタープライズ設計者と[Office 365](https://technet.microsoft.com/library/dn127064(v=office.14).aspx)と[マイクロソフトのエンタープライズ モビリティとセキュリティ](http://microsoft.com/ems)が含まれている、他のユーザーの間で、Azure Active Directory (識別情報) の場合、Microsoft に精通している IT プロフェッショナルIntune (デバイス管理) と Azure の情報の保護 (データ保護)。

エンタープライズ モビリティとセキュリティの E5 のライセンスを持っていないそれらの組織では、すべての計画に含まれている Azure AD [基準計画の保護機能を実装するには、少なくともをお勧めします。詳細について Azure AD ライブラリに[[基準計画の保護とは](/azure/active-directory/active-directory-conditional-access-baseline-protection)記事を参照できます。

## <a name="three-tiers-of-protection"></a>保護の 3 つの層

ほとんどの組織は、セキュリティとデータ保護に関して固有の要件を用意しています。 そのような要件は業種によって、また、組織内の職務によって変わります。 たとえば、法務部や Office 365 管理者は、他の部署のユーザーには必要のない、メール通信関連の追加セキュリティと情報保護管理を必要とすることがあります。

各業界では、特別な規制の独自のセットもあります。あたり、業界セグメント、またはジョブの機能は、セキュリティの 3 つの異なる層の推奨事項が提供されているすべての潜在的なセキュリティ オプションの推奨事項の一覧を提供することと、お客様のニーズの細分性に基づいて、適用可能な保護ではなく.

- **基準計画の保護**: データと同様に、id およびデータにアクセスするデバイスを保護するための最小の標準を確立することをお勧めします。多くの組織のニーズに合った強力な既定の保護を提供するこれらのベースラインの推奨事項に従うことができます。
- **機密保護**: 一部のお客様に高いレベルで保護する必要がありますまたはより高いレベルで保護するすべてのデータを必要とするデータのサブセットがあります。保護の強化をすべてに適用することができます。 または Office 365 環境内で特定のデータを設定します。アイデンティティと同等のレベルのセキュリティと機密性の高いデータにアクセスするデバイスを保護することをお勧めします。  
- **規制の厳しい**など一部の組織で高度に分類されているデータ、企業秘密、または規制対象データの量が少ない場合があります。マイクロソフトでは、id およびデバイスの保護を含む、これらの要件を満たす組織を支援する機能を提供します。

## <a name="services-and-concepts-for-identity-and-device-access-protection"></a>サービスと id とデバイスのアクセス保護の概念

マイクロソフト 365 エンタープライズが大規模な組織向けに設計されていますが、全員が創造性と作業を支援するため Office 365 エンタープライズ、Windows 10 企業、およびエンタープライズ モビリティ + (EMS) のセキュリティを統合して、安全にします。

Microsoft 365 サービスと id とデバイスのアクセスの重要な機能の概要を説明します。

### <a name="microsoft-azure-active-directory"></a>Microsoft Azure Active Directory

Azure AD は、包括的なアイデンティティ管理機能を提供します。アクセスのセキュリティ保護の推奨事項は、次の機能を使用します。

- **[セルフ サービスのパスワードのリセット (SSPR)](/azure/active-directory/authentication/concept-sspr-howitworks.md)** -管理者が制御できる複数の認証方法の検証を提供することによって安全にヘルプ デスクの介入なしにパスワードをリセットするのには、ユーザーを許可します。
- **[多要素認証 (MFA)](/azure/active-directory/authentication/concept-mfa-howitworks.md)** -MFA には、ユーザーのパスワードに加えて、Microsoft ユーザー認証システムのアプリケーション、または電話からの通知など、検証の 2 つのフォームを提供するユーザーが必要です。MFA、盗まれた id は、リスクを大幅に削減する、Office 365 環境にアクセスするために使用します。
- **[条件付きアクセス](/azure/active-directory/conditional-access/overview.md)** Azure AD は、ユーザーのログインの条件を評価し、アクセスを許可するを作成する条件付きのアクセス ポリシーを使用します。など、このガイドで説明機密性の高いデータへのアクセス用デバイスのコンプライアンスを要求するように条件付きのアクセス ポリシーを作成する方法。これにより、盗まれた id を持つハッカーが機密データにアクセスするリスクを大幅に削減します。デバイスが正常性およびセキュリティのための特定の要件を満たすためにも、デバイス上の機密データを保護します。
- **[Azure AD グループ](/azure/active-directory/fundamentals/active-directory-manage-groups.md)** ルールの条件付きアクセス、Intune とデバイスの管理と、ファイルと、組織内のサイトにもアクセス許可がユーザーや Azure AD グループへの割り当てに依存します。実装している保護のレベルに対応する Azure AD グループを作成することをお勧めします。など、経営幹部は、ハッカーの可能性の高い値ターゲットです。したがって、合理的に Azure AD グループにこれらの従業員を割り当てるし、このグループを条件付きのアクセス ポリシーとその他のより高度なアクセスに対する保護が設定されているポリシーに割り当てるです。
- **[デバイスの登録](/azure/active-directory/devices/overview.md)** デバイス id を提供する Azure AD にデバイスを登録します。この id は、ドメインに参加している、または準拠の Pc を必要とする条件付きのアクセス規則を適用して、ユーザーがサインインするときにデバイスを認証するために使用されます。このガイダンスでは、自動的にドメインに参加している Windows コンピューターを登録するのにはデバイスの登録を使用します。デバイスの登録は、Intune でデバイスを管理するための前提条件です。 
- **[Azure AD のアイデンティティ保護](/azure/active-directory/identity-protection/overview)**: Azure AD Id 保護を使用すると、組織のユーザーに影響を与えず、潜在的な脆弱性を検出し、低、中、高の記号でおよびユーザーへの自動修復のポリシーを構成します。このガイドでは、多要素認証のための条件付きのアクセス ポリシーを適用するには、このリスク評価に依存しています。このガイドには、ユーザーが自分のアカウントにリスクの高いアクティビティが検出された場合にパスワードの変更を必要とする条件付きのアクセス ポリシーが含まれています。

### <a name="microsoft-intune"></a>Microsoft Intune

[Intune](https://docs.microsoft.com/intune/introduction-intune)は、マイクロソフトのクラウド ・ ベースのモバイル デバイス管理サービスです。このガイダンスでは、Windows Intune Pc のデバイスの管理をお勧めし、デバイスのコンプライアンス ポリシーの構成をお勧めします。Intune では、デバイスが準拠しているし、条件付きのアクセス ポリシーを適用するときに使用する Azure AD にこのデータを送信するかどうかを決定します。

#### <a name="intune-app-protection"></a>Intune アプリケーションの保護

[Intune アプリケーション保護](https://docs.microsoft.com/intune/app-protection-policy)ポリシーは、管理に登録するデバイスの有無にかかわらず、モバイル アプリケーションで、企業のデータを保護するために使用できます。Intune は、Office 365 の情報を保護、従業員の生産性、および防止のデータが失われることもできますを確認します。アプリケーション ・ レベルのポリシーを実装すると、会社のリソースへのアクセスを制限し、IT 部門のコントロール内のデータを保持できます。

このガイドでは、承認されたアプリケーションの使用を強制して、ビジネス データにこれらのアプリケーションを使用する方法を決定するのには、推奨されるポリシーを作成する方法を示します。
