これらの推奨事項は、マイクロソフトのセキュリティで保護されたスコアと同様[の id は、Azure AD でスコア](https://docs.microsoft.com/en-us/azure/active-directory/fundamentals/identity-secure-score)を、揃えして、これらの組織のスコアが増加します。これらの推奨事項も、これらの[5 つの手順の id インフラストラクチャのセキュリティ保護](https://docs.microsoft.com/en-us/azure/security/azure-ad-secure-steps)を実装できます。 

マイクロソフトでは、一部の組織ではある固有の環境の要件や複雑性を理解しています。これらの組織のいずれかの場合は、開始点としてこれらの推奨事項を使用します。ただし、ほとんどの組織は、規定に従って、これらの推奨事項を実装できます。 

## <a name="intended-audience"></a>対象ユーザー

## <a name="three-tiers-of-protection"></a>保護の 3 つの層

各業界では、特別な規制の独自のセットもあります。あたり、業界セグメント、またはジョブの機能は、セキュリティの 3 つの異なる層の推奨事項が提供されているすべての潜在的なセキュリティ オプションの推奨事項の一覧を提供することと、お客様のニーズの細分性に基づいて、適用可能な保護ではなく.

- **機密保護**: より高いレベルで保護する必要のあるデータのサブセットがある一部のユーザーまたはすべてのデータをより高いレベルで保護する必要があります。保護の強化をすべてに適用することができます。 または Office 365 環境内で特定のデータを設定します。アイデンティティと同等のレベルのセキュリティと機密性の高いデータにアクセスするデバイスを保護することをお勧めします。  
- **規制の厳しい**: 一部の組織は高度に分類されるデータ量が少ない場合があります、consititutes、取引機密情報、または規制対象のデータです。マイクロソフトでは、id およびデバイスの保護を含む、これらの要件を満たす組織を支援する機能を提供します。

## <a name="services-and-concepts-for-identity-and-device-access-protection"></a>サービスと id とデバイスのアクセス保護の概念

Microsoft 365 エンタープライズでは、大規模な組織向けに設計されていますし、Office 365 のエンタープライズの Windows 10 企業、およびエンタープライズ モビリティ + セキュリティ (EMS)、するすべてのユーザーを支援するためのクリエイティブと共同作業を安全に統合します。

Microsoft 365 サービスと id とデバイスのアクセスの重要な機能の概要を説明します。

### <a name="microsoft-azure-active-directory"></a>Microsoft Azure Active Directory

Azure AD は、包括的なアイデンティティ管理機能を提供します。アクセスをセキュリティ保護するため、次の機能を使用してお勧めします。

- **[セルフ サービスのパスワードのリセット (SSPR)](/azure/active-directory/authentication/concept-sspr-howitworks)**: ユーザーに管理者が制御できる複数の認証方法の検証を提供することによって安全にし、ヘルプデスクが介入しなくてもパスワードのリセットを許可します。

- **[多要素認証 (MFA)](/azure/active-directory/authentication/concept-mfa-howitworks)**: MFA には、ユーザーのパスワードに加えて、Microsoft ユーザー認証システムのアプリケーション、または電話からの通知など、検証の 2 つのフォームを提供するユーザーが必要です。MFA、盗まれた id は、リスクを大幅に削減する、Office 365 環境にアクセスするために使用します。

- **[条件付きアクセス](/azure/active-directory/conditional-access/overview)**: Azure AD は、ユーザーのログインの条件を評価し、アクセスを許可するを作成する条件付きのアクセス ポリシーを使用します。など、このガイドで説明機密性の高いデータへのアクセス用デバイスのコンプライアンスを要求するように条件付きのアクセス ポリシーを作成する方法。これにより、盗まれた id を持つハッカーが機密データにアクセスするリスクを大幅に削減します。デバイスが正常性およびセキュリティのための特定の要件を満たすため、デバイス上の機密データを保護します。

- **[Azure AD グループ](/azure/active-directory/fundamentals/active-directory-manage-groups)**: 条件付きのアクセス規則、Intune とデバイスの管理およびファイルと、組織内のサイトにもアクセス許可がユーザーや Azure AD グループへの割り当てに依存します。実装している保護のレベルに対応する Azure AD グループを作成することをお勧めします。など、経営幹部は、ハッカーの可能性の高い値ターゲットです。したがって、合理的に Azure AD グループにこれらの従業員を割り当てるし、このグループを条件付きのアクセス ポリシーとその他のより高度なアクセスに対する保護が設定されているポリシーに割り当てるです。

- **[デバイスの登録](/azure/active-directory/devices/overview)**: デバイスの id を提供する Azure AD にデバイスを登録します。この id は、ドメインに参加している、または準拠の Pc を必要とする条件付きのアクセス規則を適用して、ユーザーがサインインするときにデバイスを認証するために使用されます。このガイダンスでは、自動的にドメインに参加している Windows コンピューターを登録するのにはデバイスの登録を使用します。デバイスの登録は、Intune でデバイスを管理するための前提条件です。 

- **[Azure AD のアイデンティティ保護](/azure/active-directory/identity-protection/overview)**: Azure AD Id 保護を使用すると、組織のユーザーに影響を与えず、潜在的な脆弱性を検出し、低、中、高の記号でおよびユーザーへの自動修復のポリシーを構成します。このガイドでは、多要素認証のための条件付きのアクセス ポリシーを適用するには、このリスク評価に依存しています。このガイドには、ユーザーが自分のアカウントにリスクの高いアクティビティが検出された場合にパスワードの変更を必要とする条件付きのアクセス ポリシーが含まれています。

### <a name="microsoft-intune"></a>Microsoft Intune

[Intune](https://docs.microsoft.com/intune/introduction-intune)は、マイクロソフトのクラウド ・ ベースのモバイル デバイス管理サービスです。このガイダンスでは、Windows Intune Pc のデバイスの管理をお勧めし、デバイスのコンプライアンス ポリシーの構成をお勧めします。Intune では、デバイスが準拠しているし、条件付きのアクセス ポリシーを適用するときに使用する Azure AD にこのデータを送信するかどうかを決定します。

#### <a name="intune-app-protection"></a>Intune アプリケーションの保護

[Intune アプリケーション保護](https://docs.microsoft.com/intune/app-protection-policy)ポリシーは、モバイル アプリケーション、デバイスの管理に登録の有無にかかわらず、組織のデータを保護するために使用できます。Intune は、Office 365 の情報を保護、従業員の生産性、および防止のデータが失われることもできますを確認します。アプリケーション ・ レベルのポリシーを実装すると、会社のリソースへのアクセスを制限し、IT 部門のコントロール内のデータを保持できます。

このガイドでは、承認されたアプリケーションの使用を強制して、ビジネス データにこれらのアプリケーションを使用する方法を決定するのには、推奨されるポリシーを作成する方法を示します。
